Разработать и оптимизировать систему управления очередью запросов с механизмом отказоустойчивости, используя принципы объектно-ориентированного проектирования. Студенты научатся применять шаблоны проектирования для повышения производительности и надежности системы.

## Постановка задачи
Необходимо реализовать систему обработки запросов с приоритетной очередью, которая обеспечивает:
- Гарантированную обработку запросов даже при сбоях
- Оптимальное использование ресурсов
- Мониторинг состояния системы

## Контракт ввода-вывода

### Входные данные:
- Поток запросов в формате JSON:
```json
{
    "request_id": "string",
    "priority": "HIGH|MEDIUM|LOW",
    "data": "string",
    "retry_count": int
}
```

### Выходные данные:
- Статус обработки каждого запроса:
```json
{
    "request_id": "string",
    "status": "SUCCESS|FAILED|RETRY",
    "processing_time_ms": int,
    "worker_id": "string"
}
```

## Требования к реализации

### Обязательные компоненты:
1. **Класс `PriorityQueue`**:
   - Реализация на основе кучи (heap)
   - Поддержка приоритетов HIGH, MEDIUM, LOW
   - Потокобезопасные операции добавления и извлечения

2. **Класс `WorkerPool`**:
   - Пул воркеров для параллельной обработки запросов
   - Динамическое масштабирование количества воркеров
   - Ограничение максимального количества одновременно обрабатываемых запросов

3. **Механизм повтора (Retry Mechanism)**:
   - Автоматический повтор неудачных запросов (до 3 попыток)
   - Экспоненциальная задержка между повторами
   - Отправка в "кладбище сообщений" после исчерпания попыток

### Ограничения:
- Максимальный размер очереди: 10 000 запросов
- Время обработки одного запроса: 50-500 мс (рандомизировано)
- Вероятность сбоя обработки: 10%
- Максимальное количество воркеров: 10

### Граничные случаи:
1. Переполнение очереди
2. Одновременный доступ к очереди из нескольких потоков
3. Внезапное завершение работы воркера

5. Невалидные входные данные
