### Сервер:

  - IP-адрес сервера (строка)
  - Порт сервера (целое число)
  - Логин и пароль (строка)
  - Путь к отправляемому файлу (строка)
- **Выходные данные:**
  - Статус операций в консоль
  - Полученные файлы в локальную директорию

1. **Протокол передачи:**

2. **Безопасность:**
   - Хеширование паролей с использованием SHA-256
   - Шифрование передаваемых данных алгоритмом AES-256
   - Генерация сессионного ключа при аутентификации

3. **Функциональность:**
   - Регистрация нового пользователя
   - Аутентификация существующего пользователя
   - Загрузка файла на сервер
   - Скачивание файла с сервера
   - Проверка целостности файла (MD5)

4. **Обработка ошибок:**
   - Некорректные учетные данные
   - Ошибки сети
   - Ошибки файловой системы
   - Нарушение целостности данных

| Формат логов | Текстовый | JSON | XML |
| Максимальный размер файла | 10 МБ | 50 МБ | 100 МБ |

### Ограничения:

# Лабораторная работа: Ускорение работы образовательной платформы с помощью распределенного кэширования

## Цель работы
Разработать высокопроизводительную систему кэширования для образовательной платформы, обеспечивающую быстрый доступ к учебным материалам и персональным рекомендациям.

## Постановка задачи
Создать клиент-серверное приложение, где серверная часть реализует распределенное кэширование учебных материалов с балансировкой нагрузки между нодами, а клиентская часть имитирует поведение студентов и преподавателей, генерируя запросы к системе.

## Контракт входных/выходных данных

### Входные данные:
1. Конфигурация сервера (JSON):
```json
{
  "nodes": [
    {"id": 1, "address": "edu-node1:3001", "capacity": 2000},
    {"id": 2, "address": "edu-node2:3002", "capacity": 2500}
  ],
  "cache_policy": "LRU,FIFO",
  "load_balancer": "round_robin"
}
```

2. Клиентские запросы (gRPC):
```
service CacheService {
  rpc GetMaterial (MaterialRequest) returns (MaterialResponse) {}
  rpc GetRecommendation (RecommendationRequest) returns (RecommendationResponse) {}
}
```

### Выходные данные:
1. Статистика производительности (JSON):
```json
{
  "total_requests": 1500,
  "cache_hits": 1120,
  "cache_misses": 380,
  "avg_response_time": 38.5,
  "throughput": 180.2,
  "active_nodes": 2,
  "node_failures": 0
}
```

## Требования к реализации

### Серверная часть:
- 2+ ноды кэширования с поддержкой LRU и FIFO политик вытеснения
- Балансировщик нагрузки с алгоритмом round_robin
- Поддержка TTL для кэшированных материалов
- Механизм обнаружения и обработки отказов нод
- Автоматическое перераспределение нагрузки при отказе ноды

### Клиентская часть:
- Генератор учебных запросов с настраиваемой интенсивностью
- Измерение времени отклика и подсчет статистики
- Обработка ошибок при недоступности нод

### Ограничения:
- Время отклика < 80 мс при 1200 RPS
- Использование памяти в пределах capacity ноды
- Отказоустойчивость при недоступности одной ноды

### Граничные случаи:
1. Пиковые нагрузки во время экзаменов
2. Одновременный доступ к популярным материалам
3. Обновление кэша при изменении учебных материалов
4. Обработка неверных идентификаторов курсов
5. Автоматическое восстановление после отказа ноды

## Критерии оценки
1. Корректность реализации (40%)
2. Производительность системы (30%)
3. Обработка ошибок и отказоустойчивость (20%)
4. Качество кода и документация (10%)

## Рекомендуемые инструменты
- Язык программирования: Python/Go/Java
- Сетевые библиотеки: gRPC
- Тестирование: pytest, locust
- Мониторинг: Prometheus + Grafana (опционально)

# Лабораторная работа 3: Оптимизация производительности сетевых приложений с использованием механизмов кэширования и балансировки нагрузки

## Постановка задачи
Создать клиент-серверное приложение, где серверная часть реализует распределенное кэширование с балансировкой нагрузки между несколькими нодами, а клиентская часть генерирует запросы и измеряет производительность системы.

  "cache_policy": "LRU",
  "load_balancing": "round_robin"
}
```

```
GET /cache?key=<ключ>

1. Статистика работы системы (JSON):
```json
{
  "total_requests": 1000,

}

   - Поддержка политик вытеснения: LRU (обязательно) и LFU
   - Механизм балансировки нагрузки (round-robin)
   - Обработка TTL (время жизни записей в кэше)

2. Клиентская часть:
   - Генератор запросов с настраиваемой интенсивностью
   - Измерение времени отклика
   - Подсчет hit/miss ratio

### Ограничения:
- Время отклика не должно превышать 100 мс при 1000 RPS
- Использование памяти на ноде не должно превышать указанный capacity
- Система должна корректно обрабатывать отказ одной из нод

### Граничные случаи:
1. Одновременный доступ к одному ключу с нескольких клиентов
2. Переполнение кэша
3. Сетевые задержки между нодами
4. Невалидные запросы
