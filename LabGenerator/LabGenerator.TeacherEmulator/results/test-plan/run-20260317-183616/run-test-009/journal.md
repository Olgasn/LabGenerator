# Лабораторная работа 3: Оптимизация производительности сетевых приложений с использованием механизмов кэширования и балансировки нагрузки

## Цель работы
Разработать и протестировать распределенную систему кэширования с балансировкой нагрузки, способную обрабатывать сетевые запросы с минимальной задержкой и максимальной пропускной способностью.

## Постановка задачи
Создать клиент-серверное приложение, где:
1. Серверная часть реализует распределенный кэш с несколькими узлами
2. Клиентская часть генерирует запросы к серверу
3. Реализовать алгоритм балансировки нагрузки между узлами кэша
4. Обеспечить отказоустойчивость при выходе из строя одного из узлов

## Контракт ввода-вывода

### Входные данные:
- Конфигурационный файл в формате JSON с параметрами:
  - Адреса и порты серверных узлов
  - Параметры кэша (время жизни записей, максимальный размер)
  - Параметры балансировщика (алгоритм выбора узла)
- Клиентские запросы в формате:
  - GET /resource/{id}
  - PUT /resource/{id} (с телом запроса)

### Выходные данные:
- Для GET-запросов: значение ресурса или код 404
- Для PUT-запросов: подтверждение сохранения (код 200) или ошибка (код 500)
- Логи работы системы с временными метками

## Требования к реализации

### Обязательные компоненты:
1. Серверная часть:
   - Минимум 3 узла кэша
   - Механизм репликации данных между узлами
   - Алгоритм консистентного хеширования для распределения нагрузки
   - Обработка отказов узлов (health check)

2. Клиентская часть:
   - Генератор запросов с настраиваемой интенсивностью
   - Поддержка различных шаблонов доступа (равномерный, zipfian)
   - Измерение времени отклика и количества ошибок

3. Балансировщик нагрузки:
   - Реализация минимум двух алгоритмов балансировки (round-robin, least connections)
   - Динамическое перераспределение нагрузки при отказе узла
   - Мониторинг загрузки узлов

### Ограничения:
- Язык программирования: Python 3.8+
- Использование только стандартных библиотек (кроме тестирования)
- Максимальный размер кэшируемых данных: 1 ГБ на узел
- Время отклика не должно превышать 100 мс при нагрузке 1000 запросов/сек

### Граничные случаи:
- Одновременный отказ нескольких узлов
- Перегрузка сети между узлами
- Попытка записи дублирующихся данных
- Запрос несуществующего ресурса
- Переполнение кэша

### Вариативность между студентами:
1. Алгоритм балансировки нагрузки (round-robin, least connections, random, hash-based)
2. Стратегия репликации (синхронная/асинхронная)
3. Алгоритм вытеснения данных из кэша (LRU, LFU, FIFO)
4. Механизм обнаружения отказов узлов (heartbeat, timeout-based)
