   - Шифрование имен файлов

Разработать и протестировать механизм балансировки нагрузки для TCP-сервера с использованием алгоритма Round Robin. Студенты должны получить практический опыт повышения отказоустойчивости и производительности сетевых приложений.

## Постановка задачи
Необходимо реализовать TCP-сервер с балансировкой нагрузки, который распределяет входящие соединения между несколькими бэкенд-серверами. Сервер должен обрабатывать одновременные подключения и обеспечивать отказоустойчивость при недоступности одного из бэкендов.

## Контракт входных/выходных данных

### Входные данные:
- Конфигурационный файл в формате JSON, содержащий:
  - Порт для прослушивания балансировщика
  - Список бэкенд-серверов (IP:порт)
  - Таймауты соединения
  - Максимальное количество одновременных соединений (100)

### Выходные данные:
- Логи работы балансировщика в консоль и/или файл
- Статистика распределения запросов по бэкендам
- Обработка ошибок при недоступности бэкендов

## Требования к реализации

### Обязательные требования:
1. Реализовать алгоритм Round Robin для распределения запросов
2. Обеспечить обработку минимум 100 одновременных соединений
3. Реализовать механизм проверки доступности бэкендов (health check)
4. При недоступности бэкенда автоматически исключать его из пула
5. Восстанавливать бэкенд в пуле после восстановления доступности
6. Вести журнал событий с указанием времени, типа события и задействованных узлов

### Ограничения:
- Использовать только стандартные библиотеки Python (socket, threading, json, time)
- Запрещено использовать готовые фреймворки для балансировки нагрузки
- Максимальное время отклика балансировщика - 100 мс
- Потребление памяти не должно превышать 50 МБ при 100 активных соединениях

### Граничные случаи:
- Все бэкенды недоступны
- Резкий рост количества подключений
- Потеря соединения во время передачи данных
- Некорректные данные в конфигурационном файле
- Одновременное восстановление нескольких бэкендов

### Вариации между студентами:
1. **Интервал health check**: 5-30 секунд
2. **Стратегия восстановления бэкендов**: автоматическая/ручная
3. **Формат логов**: текст/JSON
4. **Дополнительная метрика**: среднее время отклика бэкендов / количество ошибок

## Критерии оценки
1. Корректность работы алгоритма балансировки (40%)
2. Обработка ошибок и граничных случаев (30%)
3. Эффективность использования ресурсов (20%)
4. Качество кода и документация (10%)

## Рекомендуемые шаги выполнения
