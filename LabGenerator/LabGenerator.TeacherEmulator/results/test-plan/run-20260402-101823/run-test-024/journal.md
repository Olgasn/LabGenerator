# Лабораторная работа 1: Разработка сетевого протокола прикладного уровня

## Цель работы
Разработать и реализовать упрощённый протокол прикладного уровня для обмена структурированными сообщениями между клиентом и сервером, включая механизмы сериализации данных, установления соединения и обработки ошибок.

## Постановка задачи
Создать клиент-серверное приложение, реализующее протокол обмена сообщениями на основе TCP. Протокол должен обеспечивать надёжную передачу структурированных данных между клиентом и сервером с поддержкой различных типов команд и ответов.

## Контракт ввода-вывода

### Формат сообщения:
```
[Заголовок: 4 байта][Длина данных: 4 байта][Данные: N байт]
```
- Заголовок: ASCII-строка из 4 символов, идентифицирующая тип сообщения
- Длина данных: 32-битное целое число в сетевом порядке байтов (big-endian)
- Данные: сериализованная структура в формате JSON

### Поддерживаемые команды:
- **HELO** - установление соединения
- **DATA** - передача полезной нагрузки
- **BYE** - завершение соединения
- **ACKN** - подтверждение приёма
- **ERRO** - сообщение об ошибке

## Требования к реализации

### Общие требования:
1. Реализовать сервер, способный обрабатывать множественные подключения
2. Разработать клиентское приложение с консольным интерфейсом
3. Обеспечить корректную обработку сетевых ошибок и таймаутов

### Функциональные требования:
1. **Установление соединения:**
   - Клиент инициирует соединение командой HELO
   - Сервер отвечает ACKN при успешном подключении
   - При ошибке сервер отправляет ERRO с описанием

2. **Обмен данными:**
   - Клиент отправляет данные командой DATA
   - Данные должны содержать JSON-объект с минимум тремя полями
   - Сервер подтверждает приём данных командой ACKN
   - Реализовать обработку сообщений размером до 1 МБ

3. **Завершение соединения:**
   - Любая из сторон может инициировать разрыв соединения командой BYE
   - Вторая сторона должна подтвердить разрыв

### Технические требования:
1. Язык программирования: Python 3.8+
2. Использование стандартной библиотеки (socket, threading, json)
3. Код должен быть документирован в соответствии с PEP 257
4. Реализовать логирование всех сетевых операций

### Обработка граничных случаев:
1. Некорректный формат сообщения
2. Превышение максимального размера сообщения
3. Разрыв соединения во время передачи
4. Повторная попытка подключения к занятому порту
5. Обработка таймаутов при ожидании ответа

### Критерии оценки:
1. Корректность реализации протокола
2. Обработка ошибок и граничных случаев
3. Качество кода и документация

Разработка и исследование методов повышения производительности и отказоустойчивости сетевых приложений с использованием современных технологий балансировки нагрузки и механизмов резервирования.

## Постановка задачи
Разработать сетевое приложение, способное эффективно обрабатывать клиентские запросы в условиях высокой нагрузки и частичных отказов системы. Приложение должно демонстрировать устойчивость к сбоям и обеспечивать стабильное обслуживание клиентов.

## Контракт ввода-вывода

### Входные данные:
- Конфигурационный файл с параметрами:
  - Порт(а) для прослушивания
  - Настройки пула соединений
  - Параметры таймаутов
  - Конфигурация механизма кэширования
  - Настройки мониторинга производительности
- Входящие сетевые запросы от клиентов

### Выходные данные:
- Ответы на клиентские запросы
- Детальные логи работы приложения
- Расширенные метрики производительности
- Отчет об обработке ошибок

## Требования к реализации

### Обязательные компоненты:
1. **Балансировка нагрузки**:
   - Реализация адаптивных алгоритмов балансировки (Weighted Round Robin, Least Connections)
   - Поддержка динамической конфигурации в реальном времени
   - Расширенный мониторинг загрузки узлов

2. **Механизм отказоустойчивости**:
   - Комплексное обнаружение сбоев (health checks, heartbeat)
   - Автоматическое переключение трафика на резервные узлы
   - Восстановление после сбоев с сохранением состояния

3. **Кэширование данных**:
   - Многоуровневая стратегия кэширования
   - Адаптивное управление временем жизни кэша
   - Автоматическая инвалидация устаревших данных

4. **Мониторинг и логирование**:
   - Расширенный сбор метрик производительности (latency, throughput, error rate)
   - Детальное логирование всех критических событий
   - Визуализация состояния системы в реальном времени

### Ограничения:
- Время отклика не должно превышать 500 мс при 95-м процентиле
- Система должна выдерживать не менее 1000 RPS
- Допустимое время восстановления после сбоя - не более 30 секунд

### Граничные случаи:
- Полный отказ одного или нескольких узлов
- Резкий рост нагрузки (в 10 раз от базовой)
- Потеря сетевого соединения

4. Анализ эффективности реализованных механизмов

Разработать клиент-серверное приложение для учета складских операций и упаковать его в Docker-контейнеры с настройкой сетевого взаимодействия.

## Задание
Создайте систему, состоящую из:
1. Серверного приложения на Python (FastAPI), обрабатывающего запросы по управлению складом
2. Клиентского приложения на Python (CLI), взаимодействующего с сервером
3. Docker-контейнеров для каждого компонента

### Функциональные требования
- Сервер должен обрабатывать CRUD-операции для товаров (добавление, просмотр, обновление, удаление)
- Поддерживать операции прихода/расхода товаров
- Вести историю перемещений
- Обеспечивать поиск товаров по различным критериям

### Технические требования
- Использовать Python 3.9+ и FastAPI для сервера
- Для клиента использовать библиотеку Click для CLI
- Данные хранить в SQLite с использованием постоянного тома
- Настроить пользовательскую сеть Docker
- Реализовать healthcheck для сервера
- Обеспечить разрешение имён контейнеров в сети
- Размер итоговых образов не должен превышать 200 МБ
- Время запуска контейнеров - не более 3 секунд

### Docker-конфигурация

#### Dockerfile для сервера
```dockerfile
# Этап сборки зависимостей
FROM python:3.9-slim as builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# Этап финального образа
FROM python:3.9-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .
ENV PATH=/root/.local/bin:$PATH

# Настройка healthcheck
HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:8000/health || exit 1

# Создание директории для базы данных
RUN mkdir -p /data
VOLUME /data

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

#### Dockerfile для клиента (обновлен с многоэтапной сборкой)
```dockerfile
# Этап сборки зависимостей
FROM python:3.9-slim as builder
WORKDIR /app
COPY client_requirements.txt .
RUN pip install --user --no-cache-dir -r client_requirements.txt

# Промежуточный этап для минимизации образа
FROM python:3.9-slim as runner
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY client.py .
ENV PATH=/root/.local/bin:$PATH

# Финальный этап
FROM python:3.9-alpine
WORKDIR /app
COPY --from=runner /root/.local /root/.local
COPY --from=runner /app/client.py .
ENV PATH=/root/.local/bin:$PATH
CMD ["python", "client.py"]
```

#### docker-compose.yml
```yaml
version: '3.8'

networks:
  warehouse-net:
    driver: bridge
    ipam:
      config:
        - subnet: 172.20.0.0/24

services:
  server:
    build:
      context: .
      dockerfile: Dockerfile.server
    container_name: warehouse-server
    hostname: warehouse-server
    networks:
      warehouse-net:
        ipv4_address: 172.20.0.10
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=sqlite:////data/warehouse.db
      - LOG_LEVEL=INFO
    volumes:
      - warehouse-data:/data
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

  client:
    build:
      context: .
      dockerfile: Dockerfile.client
    container_name: warehouse-client
    hostname: warehouse-client
    networks:
      warehouse-net:
        ipv4_address: 172.20.0.20
    depends_on:
      server:
        condition: service_healthy
    environment:
      - SERVER_URL=http://warehouse-server:8000
      - LOG_LEVEL=INFO
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

volumes:
  warehouse-data:
```
