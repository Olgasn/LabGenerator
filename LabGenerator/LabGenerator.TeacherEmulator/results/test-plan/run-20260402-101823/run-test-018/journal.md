Разработать прототип сетевого приложения, демонстрирующего принципы взаимодействия клиента и сервера в компьютерных сетях. Студенты освоят базовые механизмы сетевого программирования, включая установку соединения, передачу данных и обработку ошибок.

## Постановка задачи
Создать клиент-серверное приложение, реализующее простой протокол обмена сообщениями. Приложение должно обеспечивать надежную передачу данных между клиентом и сервером с возможностью обработки различных сценариев сетевого взаимодействия.

## Контракт ввода/вывода

### Серверная часть:
- Принимает входящие соединения на указанном порту
- Обрабатывает запросы от клиентов
- Отправляет подтверждения и ответы
- Ведет лог всех операций

### Клиентская часть:
- Устанавливает соединение с сервером
- Отправляет запросы согласно протоколу
- Обрабатывает ответы сервера
- Выводит результаты взаимодействия

## Требования к реализации

### Обязательные требования:
1. Реализовать базовый протокол обмена сообщениями:
   - Формат сообщений: заголовок (4 байта) + длина данных (4 байта) + данные
   - Поддержка кодов операций (минимум 3 различных типа запросов)
   - Механизм подтверждения получения сообщений

2. Обработка ошибок:
   - Корректное завершение при разрыве соединения
   - Валидация входящих данных
   - Обработка таймаутов

3. Требования к коду:
   - Чистая архитектура с разделением на модули
   - Комментарии к ключевым функциям
   - Логирование всех значимых событий с указанием временных меток и типа события

### Ограничения:
- Язык программирования: Python 3.8+
- Использование только стандартных библиотек для сетевого программирования
- Запрещено использование готовых фреймворков (Flask, Django, etc.)

### Граничные случаи для тестирования:
1. Одновременное подключение нескольких клиентов
2. Передача пустого сообщения
3. Превышение максимального размера сообщения
4. Неожиданное отключение клиента
5. Некорректный формат сообщения

## Критерии оценки
1. Корректность работы протокола (40%)

Освоить практические навыки контейнеризации сетевых приложений, научиться создавать Docker-образы, управлять контейнерами и настраивать сетевое взаимодействие между ними. Студенты получат опыт упаковки приложения в контейнер и организации его работы в изолированной сетевой среде.

## Постановка задачи
Разработать сетевое приложение (клиент-серверную систему) и упаковать его в Docker-контейнеры. Обеспечить корректное сетевое взаимодействие между контейнеризованными компонентами системы.

## Контракт ввода-вывода

### Входные данные:
1. Исходный код сетевого приложения (клиент и сервер)
2. Конфигурационные файлы для Docker
3. Требования к сетевой инфраструктуре:
   - Минимальная пропускная способность сети: 100 Мбит/с
   - Поддержка протокола TCP/IP
   - Наличие статических IP-адресов для контейнеров
   - Требования к фаерволу: открытые порты 8080-8090

### Выходные данные:
1. Dockerfile для каждого компонента системы
2. docker-compose.yml для оркестрации контейнеров
3. Документация по развертыванию
4. Исходный код приложения с внесенными изменениями для контейнеризации

## Требования к реализации

### Обязательные компоненты:
1. **Сетевое приложение**:
   - Реализовать клиент-серверное взаимодействие по протоколу TCP/UDP
   - Обеспечить обработку минимум 7 различных команд/запросов (добавление/удаление клиента, запись на тренировку, просмотр расписания, отмена записи, просмотр информации о тренере, управление абонементами, генерация отчетов)
   - Реализовать логирование работы приложения

2. **Docker-контейнеризация**:
   - Создать отдельные Docker-образы для клиента и сервера на базе alpine
   - Настроить пользовательскую сеть Docker для взаимодействия контейнеров
   - Обеспечить передачу конфигурационных параметров через переменные окружения

3. **Оркестрация**:
   - Реализовать автоматический запуск системы с помощью docker-compose
   - Настроить проброс необходимых портов (8080 для клиента, 5000 для сервера)
   - Обеспечить правильный порядок запуска контейнеров

### Ограничения:
- Использовать только официальные базовые образы (alpine)
- Минимизировать размер итоговых образов
- Не использовать привилегированные порты (ниже 1024)
- Обеспечить изоляцию сетевого трафика между контейнерами

### Граничные случаи:
1. Обработка разрыва соединения
2. Повторная инициализация контейнеров
3. Недоступность одного из компонентов системы
4. Некорректные входные данные
5. Перегрузка сервера запросами
6. Потеря сетевого соединения
7. Ошибки аутентификации
