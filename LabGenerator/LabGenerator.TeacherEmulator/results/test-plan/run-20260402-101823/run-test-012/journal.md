# Лабораторная работа 1: Разработка протокола передачи данных

## Цель работы
Разработать и реализовать упрощённый протокол передачи данных поверх ненадёжного транспортного канала, обеспечивающий гарантированную доставку сообщений с проверкой целостности и порядковой нумерацией пакетов.

## Постановка задачи
Студентам необходимо создать программную реализацию протокола передачи данных, который будет обеспечивать надёжную передачу информации между клиентом и сервером через ненадёжный канал связи. Протокол должен обрабатывать возможные потери пакетов и их искажение.

## Контракт ввода-вывода

### Входные данные:
- IP-адрес и порт удалённого узла
- Файл для передачи (текстовый или бинарный)
- Параметры протокола (размер окна, таймауты, вероятность потери пакетов для тестирования)

### Выходные данные:
- Переданный файл на принимающей стороне
- Статистика передачи:
  - Общее количество переданных пакетов
  - Количество повторных передач
  - Время передачи
  - Процент потерянных пакетов
  - Контрольная сумма полученных данных

## Требования к реализации

### Функциональные требования:
1. Реализовать протокол с установлением соединения (handshake)
2. Обеспечить механизм подтверждения получения пакетов (ACK/NACK)
3. Реализовать алгоритм скользящего окна для управления потоком
4. Обрабатывать потерю пакетов с помощью таймаутов
5. Проверять целостность данных с использованием контрольной суммы (CRC32)
6. Обеспечить правильную сборку пакетов в исходное сообщение

### Ограничения:
- Использовать язык программирования Python 3.8+
- Не использовать готовые библиотеки для реализации протокола (только сокеты)
- Размер пакета данных не должен превышать 1024 байта
- Поддержка как текстовых, так и бинарных файлов

### Граничные случаи:
1. Передача пустого файла
2. Потеря установочного пакета (SYN)
3. Потеря подтверждения (ACK)
4. Множественные повторные передачи
5. Искажение данных в пакете
6. Превышение максимального количества попыток передачи

### Требования к коду:
1. Чистый, документированный код с комментариями
2. Логирование ключевых событий передачи
3. Модульная архитектура с разделением на клиентскую и серверную части
4. Обработка исключительных ситуаций
5. Юнит-тесты для основных компонентов

### Отчётность:
1. Исходный код программы
2. Скриншоты работы программы
3. Графики зависимости времени передачи от вероятности потери пакетов
4. Анализ эффективности протокола при различных условиях работы сети

# Лабораторная работа 1: Разработка протокола передачи данных

## Цель работы
Разработать и реализовать упрощённый протокол передачи данных поверх ненадёжного транспортного канала с использованием стека TCP/IP.

## Задание
Реализуйте клиент-серверное приложение, передающее файлы по ненадёжному каналу связи с использованием собственного протокола поверх UDP. Протокол должен обеспечивать:

1. Установление соединения (трёхэтапное рукопожатие)
2. Надёжную доставку пакетов с помощью подтверждений (ACK)
3. Обнаружение и исправление ошибок с помощью контрольной суммы CRC32
4. Управление потоком через механизм скользящего окна
5. Обработку таймаутов и повторных передач

## Требования к реализации
- Использовать Python 3.8+ с модулями socket, struct, zlib, time, random
- Максимальный размер пакета: 1024 байта
- Размер окна: 4 пакета
- Таймаут ожидания ACK: 1 секунда
- Максимальное количество попыток передачи: 5

## Проверка работы
1. Запустите сервер на указанном порту
2. Запустите клиент с указанием IP-адреса сервера, порта и имени файла
3. Проверьте целостность переданного файла с помощью контрольной суммы
4. Проанализируйте статистику передачи

## Критерии оценки
- Корректность реализации протокола (40%)
- Обработка ошибок и граничных случаев (30%)
- Качество кода и документация (20%)

Разработать и проанализировать механизмы повышения производительности сетевых приложений с использованием методов кэширования и балансировки нагрузки. Студенты получат практический опыт настройки и тестирования производительности распределенных сетевых сервисов.

## Постановка задачи
Создать распределенную систему, состоящую из клиентского приложения, балансировщика нагрузки и нескольких серверных узлов. Реализовать механизмы кэширования данных на стороне клиента и сервера, а также алгоритмы балансировки нагрузки между серверными узлами.

## Контракт ввода-вывода

### Входные данные:
- Конфигурационный файл в формате JSON, содержащий:
  - Адреса серверных узлов
  - Параметры кэширования (время жизни, максимальный размер)
  - Настройки алгоритмов балансировки нагрузки
- Поток клиентских запросов (генерируется автоматически или подается из файла)

### Выходные данные:
- Статистика производительности системы:
  - Среднее время отклика
  - Количество успешно обработанных запросов
  - Процент попаданий в кэш
  - Распределение нагрузки между серверами
  - Коэффициент использования CPU на серверных узлах
  - Количество активных соединений
- Визуализация метрик производительности
- Отчет с анализом результатов

## Требования к реализации

### Обязательные компоненты:
1. Клиентское приложение:
   - Реализация механизма кэширования ответов
   - Поддержка повторных запросов при сбоях
   - Генерация тестовой нагрузки

2. Балансировщик нагрузки:
   - Реализация минимум двух алгоритмов балансировки (Round Robin и Least Connections)
   - Мониторинг состояния серверных узлов
   - Механизм переключения при отказе узла
   - Поддержка проверки работоспособности узлов (health checks)

3. Серверные узлы (минимум 3 экземпляра):
   - Кэширование часто запрашиваемых данных
   - Поддержка состояния сессии
   - Логирование запросов
   - Проверка целостности данных с использованием CRC32

### Ограничения:
- Время отклика системы не должно превышать 500 мс при 95-м процентиле
- Коэффициент использования CPU на серверных узлах не более 80%
- Поддержка минимум 1000 одновременных соединений
- Размер кэша - не более 100 МБ на узел

### Граничные случаи:
- Отказ одного или нескольких серверных узлов
- Резкое увеличение нагрузки
- Невалидные запросы от клиентов
- Переполнение кэша
- Сетевые задержки и пакетная потеря

### Требования к тестированию:
1. Провести нагрузочное тестирование с разными профилями нагрузки
2. Сравнить производительность системы с включенным и выключенным кэшированием
3. Оценить эффективность разных алгоритмов балансировки
4. Протестировать отказоустойчивость системы при отключении узлов
5. Проверить корректность работы механизма проверки CRC32
6. Измерить производительность при максимальной нагрузке (1000 соединений)

### Требования к отчету:
- Описание архитектуры распределенной системы
- Анализ выбранных алгоритмов балансировки нагрузки
- Сравнительные графики производительности для разных конфигураций
- Анализ эффективности механизмов кэширования
- Оценка отказоустойчивости системы
- Выводы и рекомендации по оптимизации

### Технические требования:
- Язык программирования: Java 17
- Протокол: TCP
- Использование CRC32 для проверки целостности данных
- Запрещается использование готовых систем кэширования (Redis, Memcached)
- Допускается использование библиотек для нагрузочного тестирования (например, JMeter)
