Разработать и проанализировать механизмы повышения производительности и отказоустойчивости сетевого чат-сервера с использованием балансировки нагрузки, кеширования и мониторинга.

## Постановка задачи
Создать распределённый чат-сервер, способный обрабатывать высокую нагрузку входящих запросов, обеспечивая минимальное время отклика и устойчивость к сбоям. Реализовать систему балансировки нагрузки между серверами, механизмы кеширования и мониторинга.

## Контракт ввода/вывода

### Входные данные:
- Конфигурационный файл с параметрами серверного кластера
- Поток входящих HTTP-запросов (REST API)
- Параметры нагрузки (до 1000 RPS)
- Команды управления сервером (/add_server, /remove_server, /stats)

### Выходные данные:
- Ответы на запросы в формате JSON
- Детальная статистика производительности (время отклика, RPS, ошибки)
- Логи обработки запросов
- Графики нагрузки в реальном времени
- Отчеты об отказоустойчивости

## Требования к реализации

### Обязательные компоненты:
1. **Балансировщик нагрузки**:
   - Реализация алгоритма Round Robin с health checks
   - Динамическое управление серверами (добавление/удаление)
   - Автоматическое перераспределение нагрузки при сбоях
   - Поддержка до 1000 RPS

2. **Система кеширования**:
   - Кеширование частых запросов (GET /messages)
   - Политика замещения LRU с настраиваемым размером кеша
   - Время жизни кешированных данных (TTL)
   - Инвалидация кеша при изменении данных

3. **Мониторинг и логирование**:
   - Сбор метрик: время отклика, RPS, ошибки, загрузка CPU/RAM
   - Визуализация метрик в реальном времени (Grafana)
   - Логирование всех операций (ELK Stack)
   - Автоматические алерты при аномалиях

### Ограничения:
- Время отклика ≤ 500 мс при 95-м процентиле
- Поддержка ≥ 1000 RPS
- Автоматическое восстановление при сбоях
- Максимальный размер кеша: 1 ГБ

### Граничные случаи:
- Отказ всех серверов в кластере
- Резкий рост нагрузки (в 10 раз)
- Некорректные запросы и атаки
- Потеря соединения с базой данных
- Восстановление после сбоя

### Технические требования:
- Язык: Java 17+
- Фреймворки: Spring Boot, Micrometer
- Балансировщик: Nginx или HAProxy
- Кеш: Redis или Caffeine
- Мониторинг: Prometheus + Grafana
- Документация: OpenAPI, Javadoc
- Тестирование: JUnit 5, JMeter, 80% покрытие

### Критерии оценки:
1. Корректность реализации балансировки и отказоустойчивости (30%)
2. Производительность под нагрузкой (1000 RPS) (25%)
3. Качество реализации кеширования и политик замещения (20%)
4. Полнота мониторинга и визуализации (15%)
5. Документация и тестовое покрытие (10%)
