    private Map<String, User> users = new ConcurrentHashMap<>();

            throw new IllegalArgumentException("Book cannot be null");

    public boolean borrowBook(String isbn, String userId) {
        if (isbn == null || userId == null) {
            throw new IllegalArgumentException("ISBN and User ID cannot be null");
        }
        
        User user = users.get(userId);
        if (user == null) {
            throw new IllegalArgumentException("User not found");
        }
        
        Book book = books.get(isbn);
        if (book != null && book.isAvailable()) {
            synchronized (book) {
                if (book.isAvailable()) {
                    book.setAvailable(false);
                    user.getBorrowedBooks().add(book);
                    return true;
                }
            }
        }
        return false;
    }
    
    public boolean returnBook(String isbn, String userId) {
        if (isbn == null || userId == null) {

        }

        if (user == null) {
            throw new IllegalArgumentException("User not found");
        }
        
        Book book = books.get(isbn);
        if (book != null && !book.isAvailable()) {
            synchronized (book) {
                if (!book.isAvailable() && user.getBorrowedBooks().contains(book)) {
                    book.setAvailable(true);
                    user.getBorrowedBooks().remove(book);
                    return true;
                }
            }
        }
        return false;
    }
    
    public List<Book> searchBooks(String query) {

    }

Разработать и оптимизировать систему управления больничными ресурсами, обеспечивающую высокую производительность при обработке данных о пациентах, врачах и медицинских процедурах, а также устойчивость к сбоям.

## Постановка задачи
Создать систему управления больницей, способную эффективно обрабатывать потоки данных о пациентах, назначениях врачей, доступности палат и медицинского оборудования. Система должна обеспечивать бесперебойную работу в условиях высокой нагрузки и возможных сбоев, включая механизмы контрольных точек и восстановления состояния.

## Контракт ввода-вывода

### Входные данные:
- Поток данных о пациентах (регистрация, диагнозы, назначения)
- Расписание врачей и доступность оборудования
- Запросы на госпитализацию и выписку
- Конфигурационные параметры системы (настройки кэширования, политики восстановления, лимиты ресурсов)
- Команды управления (старт/пауза/остановка обработки, изменение параметров в runtime)

### Выходные данные:
- Распределение пациентов по палатам
- График работы врачей и оборудования
- Медицинские карты пациентов
- Статистика загруженности отделений
- Журнал событий и ошибок с временными метками
- Визуализация метрик производительности (графики загрузки, времени отклика, использования ресурсов)

## Требования к реализации

### Функциональные требования:
1. Реализовать систему управления с поддержкой:
   - Параллельной обработки запросов на госпитализацию
   - Кэширования данных о доступности палат и врачей
   - Контроля использования ресурсов
   - Механизма контрольных точек (checkpointing) для сохранения состояния системы

2. Обеспечить отказоустойчивость через:
   - Автоматическое резервирование данных и восстановление состояния из контрольных точек
   - Восстановление после сбоев оборудования
   - Обработку конфликтов расписания
   - Резервное копирование состояния системы
   - Обработку исключительных ситуаций

3. Реализовать мониторинг:
   - Загруженности отделений
   - Эффективности использования ресурсов
   - Времени обработки запросов
   - Сбор метрик производительности (время выполнения, использование памяти, пропускная способность)
   - Визуализацию статистики в реальном времени

### Нефункциональные требования:
1. Производительность:
   - Минимальное время отклика при поиске свободных палат
   - Оптимальное распределение нагрузки между отделениями
   - Эффективное использование ресурсов
   - Масштабируемость решения

2. Надежность:
   - Гарантированное сохранение данных о пациентах
   - Устойчивость к пиковым нагрузкам
   - Защита от утечек памяти
   - Устойчивость к некорректным входным данным

3. Качество кода:
   - Соблюдение принципов SOLID и ООП
   - Использование шаблонов проектирования
   - Модульное тестирование критических компонентов
   - Четкая документация API
   - Читаемость и документированность кода

### Ограничения:
- Язык программирования: Java или C#
- Использование стандартных средств параллелизма
- Конфигурируемые лимиты ресурсов
- Максимальный объем используемой памяти должен быть конфигурируемым
- Корректная обработка прерываний

### Граничные случаи для тестирования:
1. Обработка пустого входного потока
2. Одновременная регистрация множества пациентов
3. Отказ оборудования в критический момент
4. Конфликтующие назначения врачей
5. Переполнение отделения
6. Потеря соединения с базой данных
7. Превышение лимита памяти
8. Некорректные или поврежденные данные
9. Одновременный доступ к общим ресурсам
10. Длительные операции обработки
11. Внезапное завершение работы приложения

### Критерии оценки:
1. Корректность реализации всех требований (40%)
2. Качество и эффективность кода (30%)
3. Полнота тестового покрытия (20%)
